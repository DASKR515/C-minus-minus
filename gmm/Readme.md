# gmm
 * Note that you need to install `ghc` and `cc` on Linux.

**A standalone toolchain for writing and compiling C-- (Cmm) without the GHC Haskell Runtime System.**

`gmm` wraps GHC's Cmm front-end and a plain system linker to produce native ELF binaries that are free of the Haskell RTS — no `stg_*` symbols, no `libHSrts`, no multi-megabyte "Hello World" binaries. It ships with `libgmm/`, a bundled standard-library layer (`stdc--.h`, `cmmath.h`) embedded directly into the `gmm` binary, so there's nothing extra to download or configure.

```
gmm -o app main.cmm
```

That's it. One binary, one command, no separate headers to manage.

---

## What is C-- (Cmm)?

C-- (pronounced "C minus minus", written as **Cmm** in GHC's own tooling) is a low-level, C-like programming language designed by Simon Peyton Jones and Norman Ramsey in the late 1990s. The original goal was ambitious: a **portable assembly language** — a common target that any high-level compiler could generate, and any code-generator backend could consume, without either side needing to know about the other.

That original, compiler-agnostic version of C-- never really took off outside its own research project (Quick C--, abandoned around 2013). What *did* survive is a fork of it: **Cmm**, the intermediate representation GHC uses internally to compile Haskell down to machine code. GHC turns Haskell into Core, then STG, then Cmm, and from Cmm it can generate native assembly directly, hand off to LLVM, or (historically) emit plain C.

A minimal Cmm program looks like this:

```c
#include "Cmm.h"

section "data" {
    msg: bits8[] "Hello, World!\0";
}

export main;

main() {
    foreign "C" puts(msg "ptr");
    foreign "C" exit(0);
}
```

A few things worth knowing if you've never seen Cmm before:

- **`section "data" { ... }`** declares static data — strings, constants — much like a `.data` section in assembly.
- **Types are explicit bit widths**, not C-style `int`/`float`: `bits8`, `bits32`, `bits64`, `float64`, or the platform-native word size `W_`. There is no hidden implicit conversion.
- **`foreign "C" func(...)`** is how Cmm calls into the standard C ABI — this is the *only* way to reach libc functions like `puts`, `malloc`, or `sqrt`. Cmm has no built-in standard library of its own.
- **No plain `return` from `main` the way you'd expect** — see the warning further down. Cmm's calling conventions were designed around GHC's own runtime, and `main()` is a special case.
- Cmm has no classes, no generics, no memory safety — it's intentionally close to assembly, just readable.

Cmm is normally *generated by GHC itself*, not written by hand. This project (and `gmm`) exists to make hand-written Cmm practical — usable as a small, C-like systems language in its own right, without requiring you to write any Haskell or drag in the Haskell Runtime System.

## Why gmm exists

C-- (Cmm) is the low-level intermediate language used internally by the Glasgow Haskell Compiler. It's a clean, C-like language — but writing it by hand and compiling it the "normal" way (`ghc -no-hs-main file.cmm -o app`) comes with a hidden cost: **GHC statically links the Haskell Runtime System into your binary even if your program never touches Haskell.**

We measured this directly.

### GHC vs gmm — binary size

| Program | `ghc -no-hs-main` | `gmm` | Difference |
|---|---:|---:|---:|
| `main.cmm` (`puts` only) | 1,561,312 bytes | 5,768 bytes | **~270×** |
| `main.cmm` + `stdc--.h` | 9.7 MB | 16 KB | **~620×** |

The gap doesn't shrink as the program grows — it widens. That's the RTS footprint scaling in, not the program itself.

### What's actually in that "hidden" weight

Inspecting a GHC-linked binary with `strings app | grep -i stg` turns up dozens of internal RTS symbols that have nothing to do with the program logic, for example:

```
stg_stack_underflow_frame: unsupported register
stg_dead_thread entered!
stg_sig_install: bad spi
stg_ap_p_ret
...
```

And `readelf -d app | grep NEEDED` on a stock GHC build shows dynamic dependencies that a hand-written Cmm "Hello World" has no logical reason to need:

```
libgmp.so     <- GHC's Integer implementation
libffi.so     <- GHC's foreign-call/closure machinery
libiconv.so   <- GHC's text encoding layer
```

None of this is used by the program. It's there because GHC's default *linking* behavior assumes every binary it produces might need the full Haskell runtime — `-no-hs-main` only removes the Haskell entry point, not this assumption.

### How gmm avoids it

`gmm` splits compilation and linking into two separate steps instead of letting `ghc` do both:

```bash
ghc -c -no-hs-main -I<libgmm-path> main.cmm -o main.o   # compile only
cc main.o -o app -lm -lc                                 # link with a plain system linker
```

Because `cc` has no notion of the Haskell RTS, it links only what the object file actually references. The result is a binary with the same behavior and none of the extra weight.

## Installation

```bash
go build -o gmm main.go
sudo mv gmm /usr/local/bin/
```

Requires `ghc` and `cc` to be available on `PATH`. `gmm` checks for both before running and fails with a clear error if either is missing.

`libgmm/` (containing `stdc--.h`, `cmmath.h`) is embedded into the `gmm` binary at build time via Go's `embed` package — it is extracted to a temp directory automatically at runtime, so end users never need to fetch it separately.

## Usage

```bash
gmm -o app main.cmm                    # compile and link
gmm -o app main.cmm --keep-obj         # keep the intermediate .o file
gmm -o app main.cmm --libs=pthread,m   # link extra libraries (e.g. for threads.h)
gmm --help                  
```

> **Note the flag order:** `-o` comes *before* the input file (`gmm -o app main.cmm`), not after like `gcc file.c -o app`. This isn't a style choice — Go's `flag` package stops parsing flags at the first non-flag argument it sees. If the input file came first, anything after it (like `-o app`) would be silently ignored instead of parsed. Putting `-o` first avoids that trap entirely.

### Example

```c
#include "Cmm.h"
#include "stdc--.h"

section "data" {
    msg: bits8[] "Hi Jordan\0";
}

export main;

main() {
    mout(msg "ptr");
    foreign "C" exit(0);
}
```

```bash
gmm -o app main.cmm && ./app
# Hi Jordan
```

### gmm vs. plain ghc — same source, two workflows

| | `ghc -no-hs-main main.cmm -o app` | `gmm -o app main.cmm` |
|---|---|---|
| Command | 1 (GHC compiles *and* links) | 1 (gmm runs GHC then `cc` internally) |
| Headers | must supply `-I` yourself | bundled, found automatically |
| Output size | multi-MB (RTS statically linked) | few KB (RTS-free) |
| `stg_*` symbols | present | absent |
| Extra deps (`libgmp`, `libffi`, `libiconv`) | pulled in | not pulled in |
| `cmmath.h` warning | none | printed automatically |

**Recommendation: always prefer `gmm` over calling `ghc` directly for standalone Cmm programs.** Plain `ghc -no-hs-main` still works and produces a *correct* binary — it's what `gmm` uses under the hood for the compile step — but it silently links the full Haskell RTS on top, every single time, for no benefit if your program never touches Haskell. `gmm` gives you the identical compiled program with none of that dead weight, plus header discovery and safety warnings for free. There's no downside to using it and a large one to skipping it.

> ⚠️ **Use `foreign "C" exit(0);` to terminate `main()`, not `return (0);`.**
> A bare Cmm-level `return` from `main` follows GHC's STG calling convention, which expects a stack frame the RTS normally sets up. Without it, the program can crash (segfault) *after* correct output has already been printed. This happens with plain `ghc -no-hs-main` too — it is not specific to `gmm`. `gmm` will warn if it detects `return` in `main()` without a preceding `exit()` call.

## Bundled libraries (`libgmm/`)

| File | Status | Notes |
|---|---|---|
| `stdc--.h` | Semi-stable | FFI wrapper macros for `stdio.h`, `stdlib.h`, `string.h`, `time.h`, `ctype.h`, `threads.h`, `stdatomic.h`, and more. |
| `cmmath.h` | ⚠️ **Experimental** | Wraps `math.h`. Known to be unstable — do not rely on it for production code. `gmm` prints a warning if a file includes it. |

If your file includes `cmmath.h`, `gmm` will print:

```
warning: cmmath.h is experimental and may produce incorrect results. Use with caution.
```

## What gmm is *not*

- Not a new compiler front-end — Cmm parsing and code generation still come entirely from GHC.
- Not a replacement for GHC — you still need GHC installed.
- Not a guarantee of RTS-free output in every case — some standard library calls may pull in dependencies of their own (e.g. `-lpthread` for `threads.h`). Always verify with `readelf -d` / `strings` for your specific use case.

## Verifying the claims yourself

```bash
# Size
ls -la app

# RTS symbols (should be empty for gmm output)
strings app | grep -i stg

# Dynamic dependencies (should be minimal — libc, libm — for gmm output)
readelf -d app | grep NEEDED
```
